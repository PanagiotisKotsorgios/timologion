════════════════════════════════════════════════════════════════
  timologion — ONE-CLICK COOLIFY DEPLOYMENT
════════════════════════════════════════════════════════════════

The repo ships with a full docker-compose.yaml that Coolify runs
end-to-end. The database, migrations, admin user, secrets — all
happen on the first "Deploy" click. You don't have to provision
MySQL separately or generate any secrets by hand.


──── STEP 1 · CREATE THE APPLICATION IN COOLIFY ──────────────────

  Coolify → Resources → New Resource → Application
    → Public Repository
    → Docker Compose

  Fill in:
    Repository URL:  https://github.com/PanagiotisKotsorgios/timologion
    Branch:          main
    Base directory:  /
    Docker Compose location:  docker-compose.yaml

  Coolify will parse the compose file and detect two services:
     - "app"    (the Next.js app)
     - "mysql"  (the MySQL 8 database, private volume attached)

  Save.


──── STEP 2 · (OPTIONAL) INITIAL ADMIN VARIABLES ─────────────────

  If you want your super_admin user auto-created on the very first
  boot, add these three variables in Coolify → Environment Variables
  BEFORE the first deploy:

    INITIAL_ADMIN_EMAIL=you@yourcompany.gr
    INITIAL_ADMIN_PASSWORD=       (leave empty for a random one)
    INITIAL_ADMIN_NAME=Your Name

  If you leave INITIAL_ADMIN_PASSWORD blank, the first-boot script
  generates a random 32-char password and PRINTS it to the container
  logs surrounded by an ASCII box. Copy it immediately — that's the
  only place it will ever appear.

  If you skip these variables entirely, no admin is created and you
  can promote yourself later via the terminal:

    Coolify → app → Terminal
    $ npx tsx scripts/admin.ts promote you@yourcompany.gr super_admin


──── STEP 3 · CLICK "DEPLOY" ─────────────────────────────────────

  That's it. Coolify will:

    1. Pull the repo.
    2. Auto-generate MYSQL passwords, SESSION_SECRET, CRON_SECRET.
       (These persist across redeploys — set once, live forever.)
    3. Start the MySQL container with a persistent volume.
    4. Build the Next.js image from the multi-stage Dockerfile.
    5. Wait for MySQL to be healthy.
    6. Start the app container. Inside it:
         a. npx prisma migrate deploy    (creates all tables)
         b. npx tsx scripts/first-boot   (seeds 4 default plans,
                                           creates initial admin
                                           if configured)
         c. npm run start                 (boots Next.js)
    7. Attach the app's public FQDN to the service and issue
       a Let's Encrypt cert automatically.

  Watch the build logs. When you see "Ready in Xms" the app is live.

  First deploy: ~5 min. Subsequent deploys: ~1 min.


──── STEP 4 · HTTPS + DOMAIN ─────────────────────────────────────

  In Coolify → your app → Domains:
    Add your domain (e.g. app.timologion.gr).
    Enable "Force HTTPS" and Let's Encrypt.
    Point your DNS A record at Coolify's server IP.

  IMPORTANT: after adding the domain, re-deploy so APP_BASE_URL
  picks up the new HTTPS domain. Then all email links and OAuth
  redirects will point to the correct host.


──── STEP 5 · WIRE THE CRON JOBS ─────────────────────────────────

  The app has four scheduled endpoints protected by CRON_SECRET
  (auto-generated in step 3 — you can copy it from the app's
  Environment Variables tab).

  In Coolify → your app → Scheduled Tasks, add four entries.
  All use the same command template:

    curl -sS -H "Authorization: Bearer $CRON_SECRET" \
         https://your-domain.com/api/cron/<name>

  ┌──────────────┬────────────┬────────────────────────────┐
  │ Endpoint     │ Schedule   │ What it does               │
  ├──────────────┼────────────┼────────────────────────────┤
  │ recurring    │ 0 6 * * *  │ Materializes repeating     │
  │              │            │ documents into drafts      │
  ├──────────────┼────────────┼────────────────────────────┤
  │ reminders    │ 0 8 * * *  │ CRM task due-date          │
  │              │            │ notifications              │
  ├──────────────┼────────────┼────────────────────────────┤
  │ low-stock    │ 30 8 * * * │ Low-inventory alerts       │
  │              │            │ to owner/admin             │
  ├──────────────┼────────────┼────────────────────────────┤
  │ unpaid-      │ 0 9 * * 1  │ Weekly Monday morning      │
  │ digest       │            │ unpaid-invoices summary    │
  └──────────────┴────────────┴────────────────────────────┘


──── STEP 6 · EMAIL DELIVERY (BREVO) ─────────────────────────────

  Log in as your super_admin at:
    https://your-domain.com/admin/email

  Fill in:
    Brevo API key    (from https://app.brevo.com → SMTP & API)
    Sender email     e.g. noreply@timologion.gr
    Sender name      timologion
    Support address  support@timologion.gr

  Send a test email from the same page to confirm delivery.


──── STEP 7 · OAuth PROVIDERS (WHEN READY) ───────────────────────

  Google:
    https://console.cloud.google.com → Credentials
    → OAuth 2.0 Client → Web application
    Redirect URI: https://your-domain.com/api/auth/google/callback

  Facebook:
    https://developers.facebook.com/apps → Facebook Login → Web
    Redirect URI: https://your-domain.com/api/auth/fb/callback

  Set GOOGLE_CLIENT_ID / GOOGLE_CLIENT_SECRET / FACEBOOK_APP_ID /
  FACEBOOK_APP_SECRET in Coolify → Environment Variables and
  redeploy. The buttons on the login/register pages will start
  working immediately.


──── STEP 8 · WRAPP INTEGRATION ──────────────────────────────────

  When Wrapp gives you staging credentials:

    1. Set WRAPP_API_KEY in Coolify → Environment Variables.
    2. Redeploy.
    3. Log in as a business owner, go to
       Ρυθμίσεις → Ηλεκτρονική έκδοση and complete activation.
    4. Issue a test retail_receipt against Wrapp staging.
    5. Verify MARK / UID / QR come back on the document detail page.

  Then repeat for each DocumentType (invoice, service_invoice,
  credit_note, delivery_note, etc.).


──── AUTO-GENERATED SECRETS (FYI) ────────────────────────────────

  Coolify's magic env vars produce and persist these on first
  deploy — you never touch them:

    SESSION_SECRET             64-char, HMAC session tokens
    CRON_SECRET                bearer token for /api/cron/*
    MYSQL_ROOT_PASSWORD        healthchecks only
    MYSQL_PASSWORD             app's DB user password
    DATABASE_URL               composed automatically from above

  You can view (but not regenerate) them from Coolify →
  Environment Variables. If you ever want to rotate SESSION_SECRET,
  delete just that variable in Coolify and redeploy — a new one
  will be generated, invalidating all existing sessions.


──── OPERATIONAL NOTES ───────────────────────────────────────────

  • Prisma migrations run on every container start (idempotent).
    Deploy without fear of duplicate applications.

  • first-boot.ts also runs on every start but only does work
    when the DB is empty. Safe to trigger.

  • App runs as non-root user (uid 1001) inside the container.

  • The MySQL data lives in the `mysql-data` named volume.
    Coolify auto-includes it in its backup schedule when you
    enable backups on the resource.

  • Logs are structured JSON from src/lib/logger.ts — grep with
    fields like `msg`, `level`, `businessId`, `documentId`.


──── ROLLBACK ────────────────────────────────────────────────────

  Coolify → your app → Deployments → pick a previous green
  deploy → "Redeploy". Prisma will happily migrate forward again
  on the next run.

  For DB rollback: restore the mysql-data volume from Coolify's
  last automatic backup, then redeploy the app image that
  corresponds to that schema state.


──── UPDATING ─────────────────────────────────────────────────────

  Push to `main` on GitHub. If you enabled Coolify's automatic
  deployment webhook, it re-deploys automatically. Otherwise
  click "Deploy" manually in the Coolify UI.
